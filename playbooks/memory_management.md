# Memory management — структура и правила обновления

Память команды — это **индекс + файлы**. `MEMORY.md` хранит только ссылки; каждый урок/решение/наблюдение — отдельный markdown в `memory/`.

## Зачем

- Новая сессия не знает контекста предыдущих, если ничего не сохранено.
- Урок, забытый после инцидента, повторяется (повторное расследование, повторная ошибка).
- Без индекса агенты не находят релевантную память, даже если она есть.

## Таксономия файлов

| Префикс | Тип | Когда писать |
|---------|-----|--------------|
| `feedback_*` | Правило/предпочтение от пользователя или урок из ошибки | Юзер явно поправил поведение, или повторилась ошибка |
| `project_*` | Состояние/решение/инфра проекта | Архитектурный выбор, статус фичи, описание инфры |
| `user_*` | Личные особенности пользователя/окружения | «у юзера есть ADB локально», «работает с такого-то VPN» |
| `reference_*` | Справочник (cheat-sheet, рецепты) | Часто переиспользуемая последовательность команд |

## `MEMORY.md` как индекс

Каждая строка — одна ссылка с одним коротким описанием. Не содержание, только указатель:

```
- [Заголовок](filename.md) — одна фраза почему это важно
```

См. `templates/memory_index.md` как стартовый skeleton.

## Когда писать новый файл vs обновлять существующий

**Новый файл**, если:
- Новый класс проблемы / новая зона ответственности.
- Урок не вписывается ни в один существующий.
- Тема может разрастись (тогда лучше отдельно с начала).

**Обновлять существующий**, если:
- Дополнение к уже сформулированному правилу (новый кейс, новый пример).
- Мелкое уточнение (опечатка, новая команда).
- Файл — журнал (например, `project_resource_tracking.md` пополняется записями).

## Когда обязательно писать в память

Канонический список триггеров — `scenarios/post-incident-memory-update.md`.

## Front-matter (рекомендуется)

```
---
name: Краткое название (одна строка)
description: Однопредложенная аннотация
type: project | feedback | reference | user
originSessionId: <id> (опционально)
---
```

## Шаблон тела файла

```
**Правило/Наблюдение:** одна фраза, суть.

**Why:** один-два кейса из практики, дата если важно.

**How to apply:** пошагово как применить.
```

## Cleanup

Ритуал очистки — `scenarios/post-incident-memory-update.md` раздел «Cleanup ритуал».

## Session start ритуал

Полный протокол — **`scenarios/session-start.md`** (Шаг 0: проверка модели → Шаг 1: загрузка всей памяти).

Краткая версия: проверить модель (свежая + 1M ctx) → Read `MEMORY.md` → все файлы батчем → `project_active_plan.md` первым если есть.

## Отслеживание активного плана

Если начата нетривиальная задача (3+ шага, несколько агентов) — создать или обновить `memory/project_active_plan.md` с:
- Текущий шаг и его статус
- Что сделано / что остаётся
- Следующее конкретное действие

При resume сессии — читать этот файл первым.
