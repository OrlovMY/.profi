# Standard: документация — чинить пробел на месте (fix-on-miss)

**Статус:** обязательный · **Введён:** 2026-06-01 (owner) · **Применимость:** все проекты, все роли

## Правило

Если информация, которая **по своей природе должна быть в документации** (паспорт фичи, README, OpenAPI, naming-таблица, карта серверов и т.п.), не находится в ней **однозначно за один взгляд** — это дефект документации. Дописать недостающее в документацию **сразу в тот же ход**, не откладывая.

«Должна быть в документации» = факты, которые понадобятся снова: канонические имена (бинарей, APK, пакетов, файлов, слотов публикации), URL/эндпоинты, пути на сервере, версии, контракты, владельцы, флаги.

## Doc-home таксономия (ГДЕ что хранится = где искать первым = куда дописывать)

Если пришлось **читать исходники/делать предположения**, чтобы ответить на вопрос ниже — значит факт лежит не там, где должен. Дописать в соответствующий doc-дом:

| Тип факта | Где должно жить (doc-home) | Пример вопроса |
|---|---|---|
| Канонические **имена** (бинарь, APK, пакет, applicationId, слот публикации, контейнер) | **паспорт фичи/компонента** (`docs/passports/<area>/...`) + таблица «слоты/legacy» | «как называется CPE-APK?» |
| **Как работает механизм** (self-update, enrollment, checkin, capture, очередь) | **паспорт механизма** — пошаговый flow + кто кого вызывает | «CPE обновляется через слот или каталог?» |
| **Контракты/инварианты** (version_code схема, формат payload, обязательные поля, endianness) | паспорт + раздел «Invariants» | «какой version_code у 0.1.38 и почему 138?» |
| **API endpoints** | `docs/openapi.yaml` (+ ссылка из паспорта) | «какой роут у control-stream?» |
| **Пути/хосты/слоты на сервере** | паспорт деплоя / CLAUDE.md «карта сервера» | «куда публикуется APK?» |
| **Версии в проде сейчас** | каталог `agent_versions` (БД) — источник истины рантайма; паспорт описывает СХЕМУ | «какая версия finish сейчас?» |
| **Процессные правила, роли, плейбуки** | `.profi/` (knowledge/standards, roles, playbooks) | «как правильно деплоить?» |

**Принцип:** первое место, КУДА смотришь за фактом = первое место, КУДА его дописываешь, если не нашёл. Если для ответа «как работает X» пришлось открыть .go/.java/.tsx — это doc-gap по строке «как работает механизм»: написать паспорт механизма в тот же ход.

## Триггер (как понять, что сработало правило)

- Пришлось искать grep'ом/по коду то, что должно лежать в паспорте/README.
- Нашёл несколько противоречивых значений (разные имена в разных местах) — зафиксировать **каноническое** + явно пометить legacy/устаревшие.
- Ответил пользователю «по факту из кода», а не «по документации».

В любом из этих случаев — **сразу** обновить соответствующий doc и сослаться на него.

## Почему

- Поиск по коду каждый раз = потерянное время и риск ответить по устаревшему/неканоничному значению.
- Документация — единственный масштабируемый источник; «знал по памяти/нашёл в коде» не переносится на команду и на будущие сессии.
- Совместимо с правилами: function passport (single source of truth), capture findings, naming contract.

## Как применять (чек-лист)

1. Определить правильный doc-дом для факта (паспорт фичи / README модуля / OpenAPI / naming-таблица).
2. Внести каноническое значение + (если есть) таблицу «где какое имя» с пометкой legacy.
3. Если doc'а нет вовсе — создать минимальный паспорт.
4. Сослаться на него в ответе пользователю.
5. Закоммитить вместе с правкой/ответом.
