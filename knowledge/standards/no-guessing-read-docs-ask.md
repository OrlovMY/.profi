# No guessing — read docs, ask assumptions (HARD)

**Запрещено гадать о поведении уже реализованного функционала. Порядок: (1) прочитать документацию проекта (паспорт/контракт/openapi), (2) при неясности — посмотреть код/живые данные, (3) если остаётся ПРЕДПОЛОЖЕНИЕ — спросить владельца, а не писать код на догадке.**

## Правило
0. **🔴 Дешёвая/бесплатная проверка существует → ВЫПОЛНИ её, не делай вывод по одному сигналу (HARD).** Если утверждение можно подтвердить локальной/бесплатной командой (другое имя бинаря, `command -v`/`which`, запрос в БД, чтение файла, один `curl`) — проверяй, а НЕ заключай. Один негативный сигнал ≠ доказательство отсутствия: одна неудачная команда/алиас не значит «нет инструмента», одна пустая выборка не значит «фичи нет». «Предположительно X» там, где X проверяем за секунды, — запрещено. Прецедент 2026-06-19: запустил `python3`, получил Windows-заглушку Store → заявил «python нет»; на деле `python`/`py` = Python 3.13 (проверка `command -v python` заняла бы секунду).
1. **Сначала документация.** Перед диагностикой/правкой существующей фичи — найти и прочитать её паспорт (`docs/passports/**`), контракт, openapi. Если факт должен быть в доке, но его нет/не находится — это дефект документации (дописать), а не повод гадать.
2. **Не выдумывать форматы/поведение.** Любое утверждение о том, «как работает X» или «какой формат Y», должно опираться на: документацию, код, или РЕАЛЬНО пойманные данные. Фразы вида «вероятно там будет …» в коде/фиксе = гадание = запрещено.
3. **Предположение → вопрос владельцу.** Если после доки+кода остаётся непроверенное допущение, влияющее на код/правку — СПРОСИТЬ владельца коротким вопросом, НЕ писать «лишний код» на догадке. Лучше уточнить, чем нагенерить неверное.
4. Отличать ДОКАЗАННОЕ (данные/док/код) от ПРЕДПОЛАГАЕМОГО — и явно это разделять в ответе.

## Почему
Гадание по реализованному = лишний/неверный код, ложные диагнозы, потеря доверия. Прецедент 2026-06-19: при разборе BLKBPL24N предположил, что код-блокировки приходят в `feature_code` sim2m (видел там только CC/CF/CH/GPRSI) — на деле BLKBPL24N в my.beeline.ru; владелец трижды указал «не гадай, читай доку, спрашивай». Доку (паспорт sim-blocked-status / operator-cabinets) надо было прочитать ПЕРВОЙ.

## Как применять
- «Есть ли паспорт/контракт на это?» → `grep docs/passports` ПЕРЕД гипотезами.
- Ловить реальные данные (как с body Beeline) вместо догадок о формате.
- Не уверен и нет доки → 1 вопрос владельцу, затем код.
