# Standard: сверка с бэклогом до создания задачи (не плодить дубли)

**Статус:** обязательный · **Применимость:** все проекты, любой момент создания задачи/таска/бэклог-пункта.

## Правило
**Перед созданием новой задачи — ВСЕГДА сверься с бэклогом/трекером.** Не дублировать. Если есть похожая (та же область/намерение, пусть иначе сформулированная) — рассмотреть **обновление и реализацию уже существующей**, а не заведение новой.

## Как применять
1. **До `TaskCreate` / записи в бэклог** — поиск по трекеру/бэклогу: ключевые слова темы, затронутые компоненты, синонимы. Прочитать кандидатов целиком (не только заголовок).
2. **Если нашёлся похожий пункт:**
   - перекрывается полностью → **не создавать новый**; при необходимости обновить описание/скоуп существующего и вести работу по нему;
   - частично (новое — это фаза/уточнение существующего) → **дополнить существующий** (под-пункт/фаза), а не плодить параллельный;
   - реально другое → создавать новый, но в описании сослаться на смежный, чтобы не разъезжались.
3. **Закрывая работу** — обновить статус именно того пункта, что вёл (не оставлять задвоенные открытыми).
4. Сомнение «дубль или нет» — это сигнал прочитать оба внимательно и при равенстве выбрать существующий (меньший ID/с историей).

## Зачем
Дубли распыляют усилия, теряют контекст/историю решения, дают противоречивые статусы и риск двойной реализации. Один пункт = один источник правды по теме.

## Связь
Усиливает [[probe-before-implement]] (бэклог врёт — проверяй РЕАЛЬНОЕ состояние) и дисциплину трекера: сначала «что уже есть по теме», потом действие.
