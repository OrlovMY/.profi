# Стандарт: тесты/верификация на КАЖДОМ этапе задачи (definition of done)

**Класс:** HARD. **Появился:** owner-директива 2026-06-04 (MDMy) — «правильный план любой задачи должен включать тесты на всех этапах».

## Принцип
Задача НЕ считается выполненной, пока на **каждом** этапе её жизненного цикла пройдена своя проверка. Тест-план определяется **в начале** (на этапе плана/паспорта), а не придумывается в конце. «Код написан» и «код выглядит правильно» ≠ done. Done = **видимый эффект подтверждён тестом** на целевой среде + регресс не внесён.

## Канонический жизненный цикл + ГЕЙТ на каждом этапе
Нельзя переходить к следующему этапу, не пройдя гейт текущего.

| # | Этап | Что проверяем (гейт) | Связанный стандарт |
|---|------|----------------------|--------------------|
| 1 | **Probe / понимание** | Воспроизвести проблему/текущее поведение НА ФАКТАХ (schema/handler/route/FE/prod), а не по backlog. Подтвердить, что проблема реальна и не сделана уже. | probe-before-implement |
| 2 | **План / паспорт** | Паспорт фичи написан ДО кода; **тут же зафиксирован тест-план** (что и как будет доказывать успех на каждом следующем этапе) + naming contract для data-pipeline. | function-passport, naming-contract |
| 3 | **Реализация** | Код в стиле окружения; ветка по git-discipline. | git-branch-discipline |
| 4 | **Сборка + unit** | Код **компилируется** (build/cross-build под целевую ОС); unit-тесты на новую логику (парсинг/гейты/генерация) — зелёные. Без unit там, где логику можно изолированно проверить — недо-готово. | — |
| 5 | **Интеграция** | Компоненты связаны; интеграционный тест: эндпоинт отвечает / флоу проходит в dev|staging (не только «функция возвращает»). | — |
| 6 | **Верификация на целевой среде** | **Видимый эффект** на реальном target: устройство/UI/API/прод (curl/grep/скрин/лог), не «код выглядит верно». Для агентов — golden-rule (install + **self-update вперёд**). Для деплоя — проверить, что задеплоен именно нужный бинарь. | verify-deploy-actually-shipped, agent-selfupdate-golden-rule |
| 7 | **Регресс** | Смежные флоу не сломаны (прогнать/проверить связанное); изменения ролей/правил только улучшают. | no-regression |
| 8 | **Capture** | Находки/паспорт/changelog/tracker/память обновлены в тот же ход. | capture-findings, doc-completeness-fix-on-miss |

## Правила
- **Тест-план — часть плана.** На этапе 2 в паспорте/задании явный раздел «Тесты по этапам»: что докажет успех на 4/5/6/7. Нельзя начинать код без него.
- **Гейт = блок.** Не прошёл гейт этапа — не идёшь дальше и не пишешь «готово». Если проверка невозможна (нет устройства/деплоя) — явно «НЕ ПРОВЕРЕНО на <этапе>», не выдавать за успех.
- **Считать видимый эффект, не diff-строки** (см. task-template).
- **Каждая ОС/платформа — отдельно** (golden-rule): то, что собралось/прошло на одной, не засчитывается за другую.
- **Невозможность тестирования фиксируется**, а не замалчивается: «компилируется, но end-to-end не прогнан — нужен живой стенд X».

## Шаблон «Тесты по этапам» (вставлять в паспорт/задание)
```
Тесты по этапам:
  Probe:        <как воспроизвёл проблему / что проверил в коде-проде>
  Unit:         <какие unit + как гонять; ожидаемый результат>
  Интеграция:   <эндпоинт/флоу + как проверить в dev/staging>
  Целевая среда: <устройство/UI/API/прод — конкретный видимый эффект + команда проверки>
  Регресс:      <смежные флоу, которые НЕ должны сломаться>
  Готово ТОЛЬКО когда все строки выше отмечены ✓ (или явно «НЕ ПРОВЕРЕНО + причина»).
```

## Acceptance (приёмка PM-01 отклоняет, если)
- В плане/паспорте нет раздела «Тесты по этапам».
- Заявлено «готово» без подтверждения видимого эффекта на целевой среде (этап 6).
- Нет unit там, где логика изолированно проверяема (этап 4).
- Регресс смежных флоу не проверен (этап 7).
- «НЕ ПРОВЕРЕНО» спрятано вместо явного указания.
