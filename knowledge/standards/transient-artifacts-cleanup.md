# Standard: оперативное хранилище — без накопления транзитных артефактов

**Статус:** обязательный · **Применимость:** все проекты; VPS/прод-серверы, диск.

## Правило
Оперативное хранилище (диск прод-сервера) держит **только оперативные данные**. Транзитные и регенерируемые артефакты **не накапливать** — удалять, как только перестали быть нужны:
- **git deploy-бандлы** (`*.bundle` для `git pull` на сервер) — удалять **сразу после успешного pull+rebuild**: каждый коммит уже в master/origin/зеркале, бандл — одноразовый транспорт.
- **build-выхлоп** (временные клоны репо, out-каталоги, распакованные сборки в `/tmp`) — удалять после сборки.
- **регенерируемые кэши** (gradle/npm/docker build cache, старые docker-образы) — это не архив; чистить prune'ом ради места, в холодное НЕ класть.

## Куда что
- **Совсем не нужно + регенерируется/дублируется** (бандлы, build-выхлоп, кэши) → **удалить**. Не архивировать.
- **Может пригодиться для диагностики** (старые логи, дампы, прошлые версии бинарей/прошивок) → **в холодное хранилище**, не в оперативное. Холодное условно бесконечное.
- **Оперативное** (БД, активно отдаваемые файлы, текущие версии) → остаётся на месте.

## Холодное vs интернет (когда что тянуть)
Холодное хранилище ≠ ускоритель. Замер (2026-06-18, Yandex WebDAV ~16.7 МБ/с vs CDN ~25 МБ/с): холодное **медленнее** хорошего CDN. Поэтому регенерируемое (deps, образы) тянуть из интернета, а холодное использовать для **сохранности/источника** (архив версий, бэкапы), не ради скорости. Переносить «всё в холодное ради скорости» — только если оно реально быстрее интернета (проверять замером).

## Деструктив — порядок
Удаление на проде — после `validate`: убедиться, что артефакт точно дубликат/регенерируем (бандл → коммит в origin; build-dir → есть в git). Активно отдаваемые файлы (каталог/enroll-URL) НЕ удалять — сломает клиентов. Спорное (прод-влияние, массовое удаление) → Big-7.

## Источник
MDMy 2026-06-18: VPS 85% диска — 901МБ git-бандлов + build-выхлоп + 9ГБ старых docker-образов. Решение Big-7: бандлы/выхлоп удалять по правилу; диагностируемое (логи/версии) → холодное (Yandex WebDAV); docker prune — отдельно ради места.
