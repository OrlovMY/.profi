# Standard: security-gate контента `.profi` (перед добавлением)

**Статус:** обязательный · **Применимость:** любое добавление в `.profi` (универсальная библиотека, потенциально шарится / может попасть в open-source).

## Правило
`.profi` — общая кросс-проектная библиотека и может оказаться вне приватного периметра. **Перед добавлением любого контента — пройти 3 теста. Не прошёл хоть один → только в приватную память проекта.**

### 1. Идентификатор-тест
Нет ли в тексте (включая примеры и «Источник»): имён хостов/URL, IP, SSH-целей, номеров инцидентов/тикетов, имён аккаунтов/email, значений env, конкретных route-путей, HWID/серийников, имён БД-таблиц/колонок конкретной системы?

### 2. Карта-атаки-тест
Не позволяет ли текст построить **privilege-map / endpoint-map / attack-surface** конкретной системы? Принцип («изолируй superadmin», «пустой префикс Fiber течёт») — ОК. Описание конкретной схемы ролей/маршрутов проекта — НЕ ОК.

### 3. Operational-playbook-тест
Не является ли контент пошаговым playbook для атаки на конкретное железо/инфру (EDL-рецепт с HWID, IP, keystore, схема хэширования паролей)? Если да — только приватная память.

## Эвристика
> «Можно ли опубликовать этот текст в open-source без последствий для безопасности?» Нет → память проекта.
> Тест: можно ли по тексту собрать словарь эндпоинтов / ролей / ключей / железа конкретной системы? Да → не в `.profi`.

**Сомнение → в память проекта.** `.profi` — только если проходит все три теста без купюр.

## Источник
SEC-02 (pentest-gate), курация .profi↔память: даже «принципы» могут косвенно палить архитектуру; EDL-рецепт с HWID = operational-playbook для физической атаки на флот → понижен в приватную память.
