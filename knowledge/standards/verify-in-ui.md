# Тестировать в реальном UI то, что работает в UI (HARD)

**Всегда когда есть возможность — проверять в реальном UI (браузере) то, что делалось и будет работать в UI.** Зелёная проверка API/БД/curl НЕ заменяет UI-проверку. Фича/фикс с UI-поверхностью считается подтверждённой только после того, как увидели её работающей в браузере.

## Правило
- Любая работа, чей результат виден/используется в UI (страница, вкладка, кнопка, график, список, форма) → после деплоя открыть в браузере и убедиться, что **рендерится и работает**.
- Проверять не только «нарисовалось», но и **Network** (нет 404/500 на нужных запросах) и **Console** (нет ошибок).
- API/curl-проверка — необходима, но НЕ достаточна: она бьёт по «правильному» URL/состоянию напрямую и скрывает баги слоя UI (неверный путь, удвоение префикса, пустое состояние, обработка ошибок, права).
- В DoD UI-задач: «проверено в UI» стоит наравне с тестами и «проверено в API».
- Если UI-проверка физически невозможна (нет доступа/окружения/харнесса) — явно написать, что проверен только API, а UI остаётся **неподтверждённым** (не выдавать за «готово»).

## Как (инструментарий)
- chrome-devtools MCP (или аналог): navigate → авторизоваться тест-аккаунтом → дойти до нужного экрана → snapshot/screenshot → проверить Network/Console.
- Особо ловить: удвоение базового префикса API в путях fetch-клиента; пустые/ошибочные состояния («Не удалось загрузить»); расхождение «API отдаёт — UI не показывает».

## Почему
API-зелёное ≠ UI-работает. Прецедент (MDMy 2026-06-22): бэкенд-фикс просмотра логов прошёл по curl (вернул данные), отрапортован «готово», но в UI вкладка показывала «Не удалось загрузить» — FE слал путь с уже-зашитым базовым префиксом → удвоение → 404. curl бил по правильному URL и баг скрыл. Поймала только проверка в браузере. Цена пропуска — ложное «готово» и баг у пользователя.

Связано: standard `task-testing-all-stages` (тесты на всех этапах), guide по verify-skill; антипаттерн — «отчитался готово по API, UI не открыл».
