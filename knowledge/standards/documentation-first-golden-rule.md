# 📘 ЗОЛОТОЕ ПРАВИЛО: документация-первой-важности (HARD)

**Документация — не менее важна, чем сама реализация. Часто важнее.**
Каждое взаимодействие и каждая функция приложения / агента / админки / сервера ДОЛЖНЫ быть описаны так, чтобы было понятно как всё работает — **в первую очередь ИИ-ассистенту** (чтобы он мог диагностировать, не перечитывая весь код), затем людям.

## Что обязано быть описано
- **Все взаимодействия между компонентами** (контракты): endpoint/канал, направление, формат запроса/ответа, auth, когда вызывается, что всегда/diff-only/периодически/по-запросу/никогда передаётся.
- **Каждая функция/возможность** компонента: что делает, по какому триггеру (команда/периодика/событие), вход/выход, формат данных, побочные эффекты.
- **Версионность:** какая версия что шлёт и что принимает; известные расхождения/каверзы формата (например «поле X не авторитетно», «logcat — массив, не строка»).
- **Где искать причину сбоя** по слоям (диагностический раздел).

## Гейт (когда писать)
- **До первой строки кода** новой фичи — паспорт функции (`docs/passports/...` + UI каталога паспортов), см. function-passport rule. Это и есть «документация-first».
- **При КАЖДОМ изменении протокола/контракта** — синхронно обновлять контракт-документ в том же PR. Расхождение код↔доки = баг (правило doc-completeness-fix-on-miss).
- **Найденный факт, который должен быть в доке, но не находится за один взгляд** — дописать сразу (doc-completeness-fix-on-miss).

## Критерий качества
Документация достаточна, если по ней **можно localize проблему без чтения исходников**: «нет логов → проверь 4ч-таймер + форму массива + 204 в логах». Если для понимания нужно лезть в код — документация неполна, дополнить.

## Почему золотое
Код без актуальной документации = чёрный ящик: каждая диагностика начинается с нуля, регрессии прячутся, версии расходятся незаметно. Документация — это то, что превращает набор реализаций в понимаемую систему. Поэтому она в DoD наравне с тестами и не «потом», а часть поставки.
