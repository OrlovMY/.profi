# Стандарт: дисциплина веток и деплоя (git)

**Класс:** HARD. **Появился:** инцидент — фикс закоммичен не на ту ветку, деплой затёр прод-фичу.

## Политика веток
1. **Разработка ВСЕГДА на отдельной ветке** (`feat/<name>` или `fix/<name>` от свежего main). Прямые коммиты в main запрещены (кроме тривиальных доков, и то лучше через ветку).
2. **Держать ≤1–2 активных dev-веток.** Не плодить. Если срочно потребовался фикс в другом месте — допустимо завести 2-ю ветку, НО обязательно ВЕРНУТЬСЯ и доделать первую.
3. **Готовое решение = merge в main** (`--no-ff`). Только после завершения+верификации (+деплоя для прод-фич). **Сразу после мержа ветка УДАЛЯЕТСЯ — локально И на ВСЕХ remote (origin + зеркала).** Это часть ритуала мержа, не «потом» (см. §0 ниже).
4. **main всегда актуален** — отражает прод. **ИНВАРИАНТ: на любом remote и локально НЕ должно быть ни одной ветки, уже влитой в main.** Существуют только: `main`, deploy-ref (напр. `vps-master`), и ≤1–2 активных НЕвлитых dev-веток. Влитая ветка, оставшаяся жить = нарушение, чистится немедленно.
5. **Перед стартом новой ветки** — закрыть/смержить текущую (или явно зафиксировать почему держим параллельно — срочность).
6. **Долгоживущую ветку** регулярно подтягивать к main (`git merge main`), иначе дрейф → деплой откатит чужое (см. инцидент ниже).

**Обоснование:** дивергентные осиротевшие ветки = источник «фикс есть в репо, но не в проде» (фикс ушёл на одну ветку, деплой потерял роут другой фичи). Единая линия main = единый источник истины, нет потери выкаченных фич.

## Проблема, которую предотвращаем
Фикс был закоммичен на feature-ветку, которая ответвилась от СТАРОГО main (до того как на main приехала другая фича — новый эндпоинт). Деплой этой ветки собрал бинарь, в котором **прод-фича отсутствовала** (роут пропал) → ошибка авторизации в проде. Симптом коварен: и фикс, и фича по отдельности «есть в репо», но НЕ в одном коммите.

## §0. Ритуал мержа = merge + delete (НЕРАЗДЕЛИМО)

«Слил в main → ветка перестала существовать». Мерж НЕ считается завершённым, пока ветка не удалена везде. Канонический ритуал (после завершения+верификации+деплоя):

```bash
git checkout main && git merge --no-ff feat/<name> -m "merge: ..."
git push origin main && git push <mirror> main      # + зеркала (github и т.п.)
# ↓ СРАЗУ, тем же заходом — удалить ветку отовсюду:
git branch -d feat/<name>                            # локально (-d сам откажет если не влита)
git push origin --delete feat/<name>
git push <mirror> --delete feat/<name>
git remote prune origin && git remote prune <mirror>
```

Исключение (ветку НЕ мержим и НЕ удаляем) — только если она **намеренно НЕ влита** и ждёт внешнего гейта (напр. on-device smoke перед merge). Такие держим, но их должно быть ≤1–2 и каждая с записанной причиной «почему ещё не влита».

### Периодический аудит инварианта (≥0 влитых веток)
Раз в сессию/после серии мержей — проверить, что влитых веток не осталось ни локально, ни на remote:
```bash
# локальные влитые (кроме main/deploy-ref):
git for-each-ref --format='%(refname:short)' refs/heads/ \
 | grep -vE '^(main|master|vps-master)$' \
 | while read b; do git merge-base --is-ancestor "$b" main 2>/dev/null && echo "MERGED(local): $b"; done
# remote влитые (по каждому remote):
git fetch --all -p -q
for r in origin <mirror>; do for b in $(git for-each-ref --format='%(refname:short)' refs/remotes/$r/ | grep -vE "/(HEAD|main|master)$"); do
  git merge-base --is-ancestor "$b" main 2>/dev/null && echo "MERGED($r): ${b#$r/}"; done; done
```
Любая строка `MERGED(...)` → удалить эту ветку (см. ритуал). Должно быть пусто.

## Правила (выполнять ВСЕГДА)

### 1. Перед любым коммитом — проверь активную ветку
`git branch --show-current` (или `git status` сверху). Коммить ровно туда, куда задумано:
- мелкий фикс/серверное изменение → на `main` (или dedicated `fix/<name>` от СВЕЖЕГО main);
- изменение в составе незавершённой фичи → на её feature-ветку.
Не коммить «куда оказался переключён» после возни с ветками/stash.

### 2. Feature-ветку ответвляй от СВЕЖЕГО main
`git checkout main && git pull && git checkout -b fix/<name>`. Если ветка живёт долго, пока main ушёл вперёд — **смержи main в неё перед деплоем** (`git merge main`), иначе деплой ветки откатит чужие прод-фичи.

### 3. Перед деплоем — убедись, что коммит содержит ВСЁ ожидаемое
Деплоить из единой линии (`main`), не из ad-hoc ветки. Проверка-чеклист перед сборкой:
- `git branch --show-current` = ожидаемая ветка;
- `git merge-base --is-ancestor <последний-задеплоенный-коммит> HEAD` → деплоим потомка, не дивергента (не теряем уже выкаченное);
- grep ключевых строк прод-фич в рабочем дереве (напр. роут/хэндлер), которые ДОЛЖНЫ быть.

### 4. Чистый деплой на сервер (не «грязный pull»)
`git pull <bundle> HEAD` падает на divergent → собирается СТАРЫЙ working-tree молча. Правильно:
```
git fetch <bundle> HEAD
git checkout --detach FETCH_HEAD
git branch -f main FETCH_HEAD && git checkout main
git reset --hard FETCH_HEAD   # убрать грязный working-tree
grep -c '<ключевая строка фичи>' <файл>   # верификация ПЕРЕД build
```

### 5. После деплоя — верифицировать что задеплоено то самое
curl/grep на проде: фича отвечает, ключевой код присутствует в развёрнутых исходниках. «git push OK» ≠ «в проде нужный бинарь».

### 6. Сведение веток
Если фикс уехал не туда — не передеплоивать обрывок: **смержить расходящиеся ветки в одну линию** (`git checkout main && git merge --no-ff <ветка>`), убедиться что объединённый коммит содержит всё, и деплоить его.

## Быстрый предполётный чеклист (перед commit→deploy)
1. `git branch --show-current` — та ветка?
2. содержит ли HEAD последний прод-коммит? (`merge-base --is-ancestor`)
3. grep ключевых строк всех ожидаемых фич в рабочем дереве;
4. деплой: fetch + detach + reset --hard + grep-верификация ПЕРЕД build;
5. после: curl/grep на проде.

Связано: deploy-bundle pitfall (detach HEAD), verify-own-deploys.
