# Standard: секреты с ограниченной областью — не в глобальном конфиге

**Статус:** обязательный · **Применимость:** любой проект с per-tenant/per-scope секретами.

## Правило
Секрет, чья область видимости ограничена (per-tenant, per-device, per-user), хранится в **scoped-хранилище** (БД с tenant_id, secret-manager), а НЕ в глобальном конфиге (`.env`, общий файл), где он доступен шире нужного.

- Пример: SSH-ключи/сертификаты устройств тенанта → таблица с `tenant_id`, не общий `.env`.
- Платформенный администратор получает к ним доступ только read-only и аудируемо (или не получает — см. `multi-tenant-superadmin-isolation`).
- Enforcement области — на уровне выдачи (кто запросил → только своё).

## Зачем
Глобальный конфиг = широкая область компрометации: один утёкший `.env` раскрывает секреты всех тенантов. Scoped-хранилище ограничивает радиус.

## Связано
- `multi-tenant-superadmin-isolation` · `test-account-secret-hygiene`.

## Источник
Реальный проект: правило при переносе device-ключей из `.env` в tenant-scoped таблицу.
