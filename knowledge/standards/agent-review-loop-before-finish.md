# Агент завершает только после approve-ревью оркестратора (HARD)

**Реализующий агент НЕ завершается и НЕ мержит сам по готовности кода.** Он отдаёт результат оркестратору на ревью и ЖДЁТ вердикт. Оркестратор отвечает либо «исправить: …» (агент правит и снова отдаёт на ревью), либо «всё ок» — и только тогда агент завершается. Цикл повторяется до approve.

## Зачем
Изменениями (в т.ч. правками по итогам ревью) занимается **агент «в контексте»** — он держит весь контекст своей реализации, дешевле и точнее правит, чем оркестратор «снаружи». Оркестратор не пишет код за агента — он ревьюит и выносит вердикт (правило delegate-to-sonnet: высокая модель = ТЗ+аудит, код пишет агент). Это устраняет два анти-паттерна: (а) агент сам объявил «готово» и ушёл с дефектом; (б) оркестратор сам дописывает/чинит код агента.

## Протокол (файловый rendezvous — SendMessage в среде может отсутствовать)
1. Агент по готовности кода: коммитит в свою ветку, затем пишет `<mailbox>/<id>.review-request` (краткое summary + список файлов + результаты сборки/тестов) и **ждёт** `<mailbox>/<id>.review-reply` длинными паузами (poll) до hard-cap. НЕ завершается, НЕ мержит, НЕ пушит origin.
2. Оркестратор ревьюит diff (+ при сомнениях — Big-7) и пишет `.review-reply`:
   - `STATUS: changes-requested` + конкретный нумерованный список правок → агент правит, коммитит, снова `.review-request` (новая итерация).
   - `STATUS: approved` → агент может завершиться (merge/деплой — по отдельному решению оркестратора/владельца).
3. Повтор до `approved`. Hard-cap итераций/времени — чтобы не висеть вечно; при упоре агент паркуется с вопросом.

В брифе КАЖДОГО реализующего агента оркестратор обязан вложить этот протокол (как и rendezvous-блок). Mechanism шарит инфраструктуру с [[agent-question-rendezvous]].

## Когда применимо
- Любая нетривиальная делегированная реализация (особенно рискованная: device-side/анти-кирпич/прод/безопасность).
- Тривиальные мелочи — на усмотрение, но дефолт — ревью-петля.

## Граница с другими правилами
- delegate-implementation-to-sonnet: код пишет агент; ЭТО добавляет — и правки по ревью тоже агент, не оркестратор.
- agent-question-rendezvous: там агент паркуется на ВОПРОСЕ/развилке; здесь — на РЕВЬЮ готового кода (всегда, не только при вопросе).
- challenge-suboptimal / confirm-understanding: ортогональны (про общение с владельцем).
