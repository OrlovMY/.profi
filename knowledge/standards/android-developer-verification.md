# СТАНДАРТ (HARD): Android Developer Verification — обязателен для каждого нового приложения

**Статус:** HARD / обязательный. Введён 2026-06-12.
**Причина:** С сентября 2026 в ряде регионов сертифицированные Android-устройства смогут
устанавливать **только приложения от подтверждённых разработчиков с зарегистрированными
package names** (Google Play Console → «Проверка разработчика Android» /
android-developer-verification). Касается ЛЮБОГО способа распространения, включая sideload
и Device Owner provisioning — не только публикации в Play Store.

## Правило

**Любое новое Android-приложение (новый `applicationId`) считается готовым к раздаче, только
когда оно зарегистрировано в Android Developer Verification и право владения ключом подтверждено.**
Это часть Definition of Done для нового пакета — наравне с самообновлением (см. золотое правило
агента) и паспортом фичи.

Запрещено выкатывать в прод/провижинить новый package id, не прошедший верификацию, после того
как требование станет enforced в целевом регионе.

## Чек-лист на КАЖДЫЙ новый package id

1. **applicationId**: только валидные Java-идентификаторы (буквы/цифры/`_`), **без дефисов**.
   Не переименовывать существующие пакеты «для красоты» — смена applicationId = новое приложение:
   не обновляется поверх установленного, ломает Device Owner binding + self-update + enroll.
2. **Play Console → Проверка разработчика Android → Названия пакетов → Зарегистрировать**:
   - поле «Название пакета» = точный `applicationId` (валидируется Google);
   - «Понятное название» = человекочитаемое имя.
3. **Добавить открытый ключ**: SHA-256 fingerprint signing-сертификата в формате colon-hex
   (`AA:BB:...`). Берётся из реально раздаваемого APK:
   `apksigner verify --print-certs app.apk | grep "SHA-256 digest"` → hex → разбить на пары через `:` в UPPERCASE.
   Если пакет исторически подписан НЕСКОЛЬКИМИ ключами (легаси миграция) — добавить все
   актуальные отпечатки (несколько ключей на пакет поддерживается).
4. **Доказательство владения приватным ключом** (визард «Подпишите и загрузите APK-файл»):
   - Google выдаёт per-package сниппет (напр. `D4WKBKGQYA4I4AAAAAAAAAAAAA`).
   - Создать файл `app/src/main/assets/adi-registration.properties`, содержимое = **ровно сниппет
     одной строкой** (без `key=`, без кавычек; формат подтверждён по
     github.com/android/security-samples/.../AndroidDeveloperVerificationAPKSigningExample).
   - Собрать APK с тем же `applicationId`, подписать **тем же приватным ключом**, что в п.3
     (раздавать этот APK не обязательно — нужен только для проверки ключа).
   - Загрузить APK в визард → «Отправить» → Android проверяет подпись и регистрирует владение.
5. **Зафиксировать** в паспорте проекта: package id, display name, SHA-256 отпечаток(и), статус
   верификации, дату.

## Ключевые ограничения

- **Регистрация = внешняя запись под идентичностью владельца аккаунта.** Делает владелец в Play
  Console (или агент в его авторизованной сессии) — с явным подтверждением на КАЖДУЮ отправку.
  Новый Google-вход в автоматизированном (CDP) браузере Google БЛОКИРУЕТ.
- **Разные ключи подписи → разные отпечатки.** Если флейворы/варианты подписаны разными ключами
  (напр. обычный keystore vs platform key), у пакета должны быть зарегистрированы все ключи,
  которыми подписан раздаваемый артефакт.
- Тип аккаунта Personal достаточен для регистрации package + верификации. (Organization нужен
  отдельно — для Verified DPC / Managed Google Play EMM.)

## Definition of Done для нового Android-приложения (сводно)
- [ ] applicationId валиден (без дефисов), не конфликтует/не переименовывает существующий
- [ ] паспорт фичи создан
- [ ] самообновление работает (золотое правило)
- [ ] package зарегистрирован в Developer Verification + публичный ключ(и) добавлены
- [ ] владение приватным ключом подтверждено (APK с adi-registration.properties загружен)
- [ ] статус и отпечатки записаны в паспорт
