# Standard: удаление legacy-совместимости — ТОЛЬКО data-driven (HARD)

**Статус:** ОБЯЗАТЕЛЬНЫЙ · **Область:** tolerant-reader / совместимость форматов во всех проектах

Это правило существует, чтобы **после старта новой сессии не натворить дичь** с удалением совместимости. Читать ПЕРЕД любым действием над legacy/compat-слоями.

## ❌ ОТ ЧЕГО ОТКАЗАТЬСЯ (НЕ делать)
1. **НЕ вырывать tolerant-reader / compat-слой «ради чистоты».** Он стоит в рантайме ~ноль (пара map-lookup) и это правильная практика (закон Постела). «Меньше строк» — не причина трогать прод.
2. **НЕ переписывать ядро (горячий путь приёма данных, колоночные контракты) ради переименования полей.** Это риск регресса по ВСЕМУ парку при нулевой выгоде по данным. Конкретно запрещено: править тело запроса/колонки горячего пути ради косметического переименования (напр. serial_number→serial).
3. **НЕ удалять алиас «на глаз»/по предположению, что им никто не пользуется.** Только по факту (см. ниже).
4. **НЕ пересобирать агентов всего парка ради косметики имён**, если данные уже корректны (их сворачивает сервер).

## ✅ КАК ПРАВИЛЬНО (единственный допустимый путь)
1. **Инструментировать срабатывания.** Каждый раз, когда сервер сворачивает legacy→канон, писать в счётчик попаданий алиасов (например, таблица с колонками `canonical, legacy_key, hit_count, last_hit_at`, UPSERT). Делается в слое нормализации алиасов.
2. **Решение об удалении — на данных.** Алиас можно удалять, ТОЛЬКО если:
   - нет строки в счётчике попаданий ИЛИ `last_hit_at < now() - 7 days`, И
   - прошло ≥1 недели с момента деплоя инструментирования (накоплены данные).
3. **Удалять по одному**, не пачкой; только из нормализатора атрибутов (rename-алиасы / derivations), **никогда** из тела запроса/колоночного контракта горячего пути.
4. После удаления — проверить, что соответствующие устройства не потеряли поле (одно устройство-образец каждой платформы).

## Запрос для проверки (раз в неделю)
```sql
-- какие алиасы ещё живы (били за последние 7 дней)
SELECT canonical, legacy_key, hit_count, last_hit_at
FROM alias_hits WHERE last_hit_at > now() - interval '7 days'
ORDER BY last_hit_at DESC;
-- кандидаты на удаление = алиасы из нормализатора (rename/derivations), которых
-- НЕТ в этом списке (никто не слал ≥7 дней).
```

## Контекст / прецедент
Однажды возникла задача «убрать legacy» в слое совместимости инвентаря. Разбор показал: данные уже верны (сервер сворачивает алиасы), удаление = риск по горячему пути + churn от пересборки агентов + payoff заблокирован офлайн-straggler'ом (старое устройство, которое всё ещё шлёт legacy-формат и долго не выходит на связь) → **net-negative**. Был выбран путь инструментирования вместо удаления. Связанные принципы: единый источник правды по схеме инвентаря; контракт именования артефактов/версий агента.
