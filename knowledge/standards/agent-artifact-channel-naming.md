# Standard: канал релиза в имени артефакта агента

**Статус:** обязательный · **Введён:** 2026-06-01 (owner) · **Применимость:** все агенты (Windows .exe, Android/CPE .apk, и др.)

## Правило

Имя файла артефакта **зеркалит `agent_versions.status`** — чтобы из листинга `/downloads/` канал был однозначен, а имя ↔ status совпадали один-в-один:

| status | суффикс имени | пример |
|---|---|---|
| `finish` (прод) | нет (чистое имя) | `MDMy-CPE-0.1.38.apk`, `MDMy-Agent-0.5.59.exe` |
| `test` | `-test` | `MDMy-CPE-0.1.39-test.apk` |
| `draft` | `-draft` | `MDMy-Agent-0.6.0-draft.exe` |

## Зачем
- Glance-clarity: видно канал без запроса в БД.
- **Один источник правды остаётся `status`** — суффикс лишь зеркало; они ОБЯЗАНЫ совпадать. Не плодим независимое состояние в имени.
- Не путать с маркерами ПОДПИСИ (`-testkey` у CPE = AOSP testkey, это про ключ, не про канал).

## Инвариант промоушна test→finish (атомарно, все три)
1. rename файла: убрать `-test`;
2. `UPDATE agent_versions SET apk_url=<новое имя>, status='finish' WHERE …`;
3. проверить URL 200 + sha256.

Пропуск любого шага = рассинхрон имя↔status↔apk_url (класс бага, на котором уже горели: слот vs версия, version_code 138 vs 39).

## version_code: ОДНА схема на платформу (контракт)

`version_code` — монотонный integer для advertise-сравнения (`catalog.version_code > device.BuildConfig.version_code` → обновить). Правила:
1. **На платформу — ОДНА схема, навсегда монотонная.** Менять схему на полпути ЗАПРЕЩЕНО — старые сборки с большими кодами «застрянут новее» нового релиза.
2. Каталожный `version_code` ОБЯЗАН РАВНЯТЬСЯ `BuildConfig`/ldflags коду самого артефакта (иначе петля: устройство ставит, рапортует другой код, advertise снова срабатывает).
3. Рекомендуемая схема: `major*10000 + minor*100 + patch` (0.5.60 → 50060; запас на 99 patch/minor). Главное — не смешивать с уже использованными диапазонами.

**Прецеденты багов (2026-06):**
- CPE 0.1.38: каталог 138 vs APK 39 → бесконечная петля. Фикс: 138→39.
- Windows: 3 схемы в истории (50–89 / 5400–5470 / 551–560). 0.5.40–0.5.47 имеют коды 5400–5470 > текущего finish 560 — латентная ловушка (пока нет устройств на 0.5.4x). Любой новый finish < 5470 не догонит застрявшую 0.5.4x-сборку.

**Уборка (низкий риск, нет устройств на затронутых):** привести стейл test/draft-entries к текущей монотонной схеме ИЛИ удалить устаревшие, чтобы `max(version_code)` среди любого статуса не превышал актуальный finish.
