# Standard: канал релиза в имени артефакта агента

**Статус:** обязательный · **Применимость:** все агенты/клиенты, распространяемые как скачиваемые артефакты (Windows `.exe`, Android `.apk`, и др.)

## Правило

Имя файла артефакта **зеркалит поле `status` записи о версии в каталоге** — чтобы из листинга директории загрузок канал был однозначен, а имя ↔ status совпадали один-в-один:

| status | суффикс имени | пример |
|---|---|---|
| прод (`finish`/`stable`) | нет (чистое имя) | `App-Client-0.1.38.apk`, `App-Agent-0.5.59.exe` |
| `test` | `-test` | `App-Client-0.1.39-test.apk` |
| `draft` | `-draft` | `App-Agent-0.6.0-draft.exe` |

## Зачем
- Glance-clarity: видно канал без запроса в БД.
- **Один источник правды остаётся `status`** — суффикс лишь зеркало; они ОБЯЗАНЫ совпадать. Не плодим независимое состояние в имени.
- Не путать с маркерами ПОДПИСИ (например, `-testkey` для AOSP testkey-сборки — это про ключ подписи, не про канал релиза).

## Инвариант промоушна test→прод (атомарно, все три)
1. rename файла: убрать `-test`;
2. обновить запись каталога версий: `url=<новое имя>`, `status=<прод>`;
3. проверить URL 200 + sha256.

Пропуск любого шага = рассинхрон имя↔status↔url (класс бага, на котором легко погореть: «слот vs версия», расхождение version_code в каталоге и в артефакте).

## version_code: ОДНА схема на платформу (контракт)

`version_code` — монотонный integer для advertise-сравнения (`каталог.version_code > устройство.version_code` → обновить). Правила:
1. **На платформу — ОДНА схема, навсегда монотонная.** Менять схему на полпути ЗАПРЕЩЕНО — старые сборки с большими кодами «застрянут новее» нового релиза.
2. Каталожный `version_code` ОБЯЗАН РАВНЯТЬСЯ коду самого артефакта (`BuildConfig`/ldflags и т.п.) — иначе петля: устройство ставит, рапортует другой код, advertise снова срабатывает.
3. Рекомендуемая схема: `major*10000 + minor*100 + patch` (0.5.60 → 50060; запас на 99 patch/minor). Главное — не смешивать с уже использованными диапазонами.

**Типичные баги (для иллюстрации класса):**
- Каталог рапортует один код, APK собран с другим (напр. 138 vs 39) → бесконечная петля обновления. Фикс: привести коды к равенству.
- Несколько схем в истории одной платформы (напр. диапазоны 50–89 / 5400–5470 / 551–560). Старые сборки с кодами из «высокого» диапазона оказываются «новее» текущего прод-релиза из «низкого» — латентная ловушка: пока на этих сборках нет устройств, она не стреляет, но любой новый прод-релиз с кодом ниже застрявшего диапазона не догонит зависшую сборку.

**Уборка (низкий риск, если нет устройств на затронутых версиях):** привести стейл test/draft-записи к текущей монотонной схеме ИЛИ удалить устаревшие, чтобы `max(version_code)` среди любого статуса не превышал актуальный прод.

## Смежное
- `[doc-completeness-fix-on-miss]` (этот же каталог) — где хранить факты.
- Контракт version_code: каталожный `version_code` == коду артефакта (`BuildConfig.VERSION_CODE`/ldflags), иначе цикл обновления.
