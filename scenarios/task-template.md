# Сценарий: Шаблон задания агенту

## Проблема

Агенты без жёсткой структуры задания часто:
- Трогают чужие файлы.
- Работают в неправильной ветке (или в master).
- Сообщают «готово», когда работа не смёрджена/не запушена.
- Пишут код, который не собирается, но заявляют успех.

## Решение — обязательный шаблон

Каждое задание от PM-01 агенту содержит:

```
Задача: <одна фраза — что сделать>

Ветка: feat/<name>   (создать от master через: git checkout master && git pull && git checkout -b feat/<name>)

НЕ ТРОГАТЬ:
  - <интеграционный файл 1>
  - <интеграционный файл 2>

ПЕРЕДАТЬ <владельцу интеграции>:
  - <что надо смёрджить в интеграционный файл — например, список новых маршрутов>

Тесты по этапам (ОБЯЗАТЕЛЬНО — см. knowledge/standards/task-testing-all-stages.md):
  Probe:         <как воспроизвёл проблему / проверил в коде-проде>
  Unit:          <unit на новую логику + как гонять>
  Интеграция:    <эндпоинт/флоу в dev|staging>
  Целевая среда: <устройство/UI/API/прод — видимый эффект + команда проверки>
  Регресс:       <смежные флоу, которые НЕ должны сломаться>
  → done ТОЛЬКО когда все строки ✓ (или явно «НЕ ПРОВЕРЕНО + причина»).

Проверка в конце (ОБЯЗАТЕЛЬНО):
  git status            # должно показать: On branch feat/<name>, working tree clean
  git log --oneline -3  # показать свои коммиты

Приложи к отчёту:
  - Хэш финального коммита.
  - Вывод git status.
  - Список изменений для handoff владельцу интеграции.

Если Bash запрещён или деплой невозможен:
  Напиши строго «ЗАДАЧА НЕ ЗАВЕРШЕНА» и верни список правок для ручного применения.
  НЕ заявляй «готово» — это считается ложным отчётом.
```

## Критерии приёмки

PM-01 **отклоняет** работу, если:
- Нет хэша финального коммита в отчёте.
- `git status` не clean.
- Работа в неправильной ветке.
- Заявлено «готово» без подтверждения верификации.
- Затронуты файлы из «НЕ ТРОГАТЬ».
- **Нет раздела «Тесты по этапам» / не пройден гейт какого-либо этапа** (probe→unit→интеграция→целевая среда→регресс), либо «НЕ ПРОВЕРЕНО» спрятано вместо явного указания (см. `knowledge/standards/task-testing-all-stages.md`).

## Принцип качества: считай видимый эффект, не количество изменений

Не считать правки или diff-строки — считать видимый эффект для пользователя или системы:
- Каждое изменение верифицировать (логи / API / UI / smoke test) — не «написал код → готово».
- Если верификация невозможна (нет доступа к устройству, нет деплоя) — явно сообщить об этом, не заявлять успех.
- Видимый эффект = работает ли feature как задумано, а не «код выглядит правильно».

## Почему ужесточение нужно

Реальный случай из практики:
Агенты без Bash-доступа правили файлы локально в изолированном worktree, но правки не синхронизировались с родительским репо. Отчёты «готово» вводили в заблуждение — код существовал только на ноутбуке одного из агентов и пропал.

После введения шаблона верификации такие случаи отлавливаются на этапе приёмки.
