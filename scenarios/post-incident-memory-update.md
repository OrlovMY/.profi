# Сценарий: Post-incident / post-session memory update

После каждой сессии и особенно после каждого инцидента HR-D обязан зафиксировать уроки в памяти. Без этого следующая сессия повторит те же ошибки.

## Триггеры обновления памяти

1. **Пользователь явно поправил поведение.** Сразу `feedback_*.md` с цитатой пользователя и «how to apply».
2. **Инцидент в проде / деплое.** `project_*_lessons.md` с root cause + fix + prevention.
3. **Найдена нетривиальная особенность** (API/SDK/OEM/browser quirk). `project_*_recipe.md` или `project_*_quirks.md`.
4. **Архитектурное решение принято.** `project_*_decision.md` с trade-offs.
5. **Новая роль / смена ролей.** `project_<role>_hired.md` + дополнение `project_team_history.md`.
6. **Resource observation.** Запись в `project_resource_tracking.md`.
7. **Изменилось правило workflow.** Править `project_workflow_scheme.md` + добавлять ссылку в `MEMORY.md` если новый файл.

## Формат записи

Front-matter + шаблон тела — `playbooks/memory_management.md`.

## Что обязательно вносить в `MEMORY.md`

- Каждый новый файл памяти = новая строка в `MEMORY.md`.
- Формат: `- [Заголовок](filename.md) — одна фраза почему важно`.
- Группа в правильную секцию (Status / Doctrine / Infrastructure / Team / Reference / Backlog).

## Что не нужно в память

- Очевидные идиомы языка.
- Однострочные stdlib-вызовы.
- Временные debug-наблюдения, которые потеряют актуальность через час.
- Дублирование уже существующих правил.

## Cleanup ритуал

Раз в N недель:
- Пройти `MEMORY.md`, удалить ссылки на устаревшие файлы.
- Объединить дубликаты feedback-файлов.
- Истёкшие `project_status_YYYY-MM-DD.md` оставить как историю, не редактировать.

## Anti-patterns

- «Запомню в голове, потом запишу» → следующий инцидент тот же.
- Огромный `notes.md` со всем подряд — не найти ничего. Разделять на отдельные файлы.
- Не обновить `MEMORY.md` после создания нового файла → файл есть, но никто не знает.
- Записать урок без «how to apply» — невозможно использовать.
