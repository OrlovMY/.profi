# No Regression — изменения только улучшают

**Строгое правило.** Любое изменение роли, стандарта, правила работы, плейбука или сценария должно только улучшать результат. Ухудшение недопустимо.

---

## Что это означает на практике

Перед внесением любого изменения в `.profi` или память проекта — ответить на три вопроса:

1. **Что конкретно улучшится?** (скорость, качество, безопасность, предсказуемость)
2. **Что может ухудшиться?** Если есть хоть один регресс — изменение не принимается в текущем виде.
3. **Есть ли способ получить улучшение без ухудшений?** Если да — только этот способ.

---

## Применяется к

- Ролям агентов (`roles/`)
- Сценариям и плейбукам (`scenarios/`, `playbooks/`)
- Стандартам качества (`knowledge/standards/`)
- Правилам работы команды
- Feedback-правилам в памяти проекта
- Шаблонам (`templates/`)

---

## Как проверить перед изменением

```
До изменения:  [описать текущее поведение / результат]
После:         [описать новое поведение / результат]
Регрессы:      [перечислить всё что может стать хуже]
Вывод:         УЛУЧШЕНИЕ / РЕГРЕСС (не принять) / НЕЙТРАЛЬНО (не нужно)
```

Если вывод — **РЕГРЕСС**, изменение останавливается. HR-D эскалирует в Big 7 с описанием trade-off.

---

## Частые ловушки

- **Упрощение ради краткости** — сокращение правила может убрать важный edge case.
- **«Это редко нужно»** — редкий случай, не покрытый правилом, становится инцидентом.
- **Переписывание вместо дополнения** — новый опыт должен дополнять старое правило, не заменять его без явного обоснования что старое было неверным.
- **Ускорение за счёт пропуска шагов** — убрать шаг верификации «чтобы быстрее» = регресс качества.

## Продвижение улучшений в стандарт

Если изменение прошло проверку (нет регрессов) и команда согласована — оно **немедленно становится стандартом**:

1. Обновить соответствующий файл в `.profi` (роль / сценарий / плейбук / стандарт)
2. Если изменение выявлено в ходе работы над проектом — зафиксировать в памяти проекта (`feedback_*.md`) со ссылкой на `.profi`
3. Сообщить HR-D — он обновляет `MEMORY.md` и при необходимости бриефинги агентов

**Триггеры для фиксации** — см. `scenarios/post-incident-memory-update.md`. Ключевые: пользователь подтвердил улучшение, Big 7 проголосовал, инцидент показал пробел в правиле.

**Правило:** улучшение, не зафиксированное в `.profi`, не является стандартом — оно существует только в голове одной сессии и будет потеряно.

---

## Исключения

Допускается изменение, которое ухудшает один параметр, если:
- Big 7 явно проголосовал за trade-off
- Пользователь дал явное одобрение с пониманием последствий
- Ухудшение задокументировано и принято осознанно (записать в изменяемый файл комментарием)
